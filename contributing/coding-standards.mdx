## Header guards

Use `#pragma once` instead of traditional `#ifndef` include guards. Every header file in Marlin uses `#pragma once` immediately after the license header.

```cpp
/**
 * Marlin 3D Printer Firmware
 * ...license header...
 */
#pragma once

#include "inc/MarlinConfig.h"
```

## Indentation

Use **2 spaces** for indentation throughout. Do not use tabs. This applies to all `.h`, `.cpp`, and `.ino` files.

```cpp
class Marlin {
public:
  static MarlinState state;
  static void setState(const MarlinState s) { state = s; }
  static bool is(const MarlinState s) { return state == s; }
  static bool isStopped() { return is(MF_STOPPED); }
  static bool isRunning() { return state >= MF_RUNNING; }
};
```

## Naming conventions

| Identifier type | Convention | Example |
|---|---|---|
| Preprocessor macros and defines | `UPPER_CASE` | `ENABLED(FEATURE)`, `BOARD_RAMPS_14_EFB` |
| Configuration options | `UPPER_CASE` | `AUTO_BED_LEVELING_UBL`, `LIN_ADVANCE` |
| Functions and methods | `camelCase` | `printingIsActive()`, `manage_inactivity()` |
| Local variables | `snake_case` or `camelCase` | `step_count`, `tempTarget` |
| Classes | `PascalCase` | `MarlinState`, `Marlin` |
| Enumerators | `UPPER_CASE` for values | `MF_RUNNING`, `MF_STOPPED` |

```cpp
// Defines: UPPER_CASE
#define FORCE_INLINE  __attribute__((always_inline)) inline
#define NO_INLINE     __attribute__((noinline))

// Function: camelCase
bool printingIsActive();
void startOrResumeJob();

// Class: PascalCase, methods: camelCase
class Marlin {
  static bool isStopped() { return is(MF_STOPPED); }
  static bool isRunning() { return state >= MF_RUNNING; }
};

// Enum values: UPPER_CASE
enum MarlinState : uint8_t {
  MF_INITIALIZING = 0,
  MF_STOPPED,
  MF_KILLED,
  MF_RUNNING,
};
```

## Macros

Marlin uses macros extensively to handle the large number of compile-time configuration options and to abstract hardware differences. Follow these conventions when writing macros:

- Macro names are always `UPPER_CASE`.
- Wrap macro arguments in parentheses to avoid operator precedence bugs.
- Use `do { ... } while(0)` for multi-statement macros to ensure safe use in `if`/`else` blocks.
- Prefer `constexpr` templates over macros where C++11 features are available.

```cpp
// Simple bit manipulation macros from src/core/macros.h
#define _BV(b)        (1 << (b))
#define TEST(n,b)     (!!((n)&_BV(b)))
#define SBI(A,B)      (A |= _BV(B))
#define CBI(A,B)      (A &= ~_BV(B))
#define TBI(N,B)      (N ^= _BV(B))

// Multi-statement macro using do-while(0)
#define SET_BIT_TO(N,B,TF) do{ if (TF) SBI(N,B); else CBI(N,B); }while(0)

// C++11 constexpr template preferred over macro for type safety
template <class V, class N> static constexpr void NOLESS(V& v, const N n) {
  if (n > v) v = n;
}
```

## C++ standard and compatibility

Marlin targets **C++11** and must remain compatible with both 8-bit AVR and 32-bit ARM toolchains. Keep the following in mind:

- C++11 features like `constexpr`, `static_assert`, range-based `for`, `auto`, and lambda expressions are available and encouraged where appropriate.
- Do not use features from C++14 or later unless guarded by a version check.
- Standard library usage is limited — `<math.h>`, `<stdio.h>`, and `<stdlib.h>` are included in core headers; use these rather than their C++ equivalents where possible.
- Keep memory usage minimal. 8-bit AVR boards (e.g. ATmega2560) have only 8 KB of SRAM and 256 KB of flash. Use `PROGMEM` and `F()` / `PSTR()` for string constants on AVR.
- Avoid dynamic memory allocation (`new`, `malloc`) in hot paths; prefer static allocation.

```cpp
// Use FSTR_P / F() for flash strings on AVR
static void kill(FSTR_P const lcd_error=nullptr, FSTR_P const lcd_component=nullptr, const bool steppers_off=false);

// IS_CONSTEXPR for compile-time checks compatible with C++14
#define IS_CONSTEXPR(...) __builtin_constant_p(__VA_ARGS__)
```

## Compiler attributes

Marlin defines several GCC attributes as macros for controlling inlining and optimization. Use these in performance-sensitive code:

```cpp
#define _FORCE_INLINE_  __attribute__((__always_inline__)) __inline__
#define  FORCE_INLINE   __attribute__((always_inline)) inline
#define  NO_INLINE      __attribute__((noinline))
#define  __Os           __attribute__((optimize("Os")))  // Optimize for size
#define  __O2           __attribute__((optimize("O2")))  // Optimize for speed
```

## Configuration comments

Configuration files use `@section` comments to group related options. When adding new configuration options to `Configuration.h` or `Configuration_adv.h`, place them within the correct section and add a descriptive comment above each option.

```cpp
// @section motion

// The number of linear motion axes (XYZ + extruders).
#define NUM_AXES 3

// @section temperature

// PID settings for the hotend heater.
#define PIDTEMP
```

## Braces and spacing

- Opening braces go on the same line as the statement.
- Always use braces even for single-line `if`/`else` bodies, unless the body is a one-liner on the same line as the condition (common in Marlin for compact expressions).
- Place a single space before `(` in control flow keywords (`if`, `for`, `while`) but not in function calls.

```cpp
// Control flow
if (condition) {
  doSomething();
}

// Compact single-line form acceptable in Marlin style
if (is_stopped) return;

// Function call — no space before (
manage_inactivity(false);
```

## Comments

- Use `//` for single-line comments.
- Use `/** ... */` for file and function documentation.
- Wrap long comment blocks with `//` lines, as seen throughout the source.
- Document the purpose of non-obvious macros, functions, and configuration options.

## End of file

All files must end with a newline character. Do not leave trailing whitespace on any line. The CI pipeline runs `make validate-lines` to enforce this.
